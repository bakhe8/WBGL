## السيناريو الأول — استيراد Excel بأسماء موردين خاطئة أو غير معيارية

### الوصف الواقعي

موظف إدخال بيانات يستورد ملف Excel أُعِدَّ يدوياً من قِبَل قسم الشراء. يحتوي:

- `شركة البنا للمقاولات` بدلاً من `شركة البناء للمقاولات والتشييد`
- `ALMAJAL` بدلاً من `Al Majal Group Company`
- `مؤسسة الاحمدي` (بدون همزة) بدلاً من المعياري

### ردّ فعل النظام

| الحالة                | الرد                            | صامت؟ |
| --------------------- | ------------------------------- | ----- |
| تطابق فازي ≥ 90%      | Auto-match، Status = ready      | نعم   |
| تطابق فازي بين 70-90% | يُعرَض كمقترح، Status = pending | لا    |
| لا تطابق              | Status = pending، يحتاج يدوي    | لا    |

**المشكلة الفعلية:**
بسبب BUG-001 (`levenshtein()` بالبايت لا الحرف)، التطابق الفازي للعربية يُنتج نتائج أقل دقة مما يُتوقع. `مؤسسة الاحمدي` vs `مؤسسة الأحمدي` (فرق الهمزة فقط) قد تفشل في تخطي عتبة 70% بسبب الخطأ، مما يُجبر الموظف على التصحيح اليدوي لكل ضمان.

### السجل الجنائي

✅ `guarantee_history` يسجل كل تعديل يدوي لاحقاً باسم الموظف
⚠️ لكن اسم المورد الأصلي الخاطئ يبقى في `raw_data.supplier_name` (غير مُعدَّل) — فجوة تاريخية

### نقاط تضخيم الخطأ

- موظف يشاهد "لا تطابق" → يكتب اسماً مختلفاً يدوياً في كل مرة → يُنشئ **aliases يدوية متعددة** لنفس المورد → `supplier_alternative_names` يتضخم بصيغ متعارضة

## السيناريو الثاني — اللصق النصي (Smart Paste) من بريد إلكتروني

### الوصف الواقعي

أمين خزينة يستلم بريداً من البنك يحتوي تفاصيل خطاب الضمان. ينسخ النص ويلصقه في `parse-paste`. النص يحتوي headers البريد الإلكتروني وتوقيع المرسل وتنسيق HTML محوَّل إلى نص.

### ردّ فعل النظام

`SmartPasteConfidenceService` يُحلِّل النص ويُنتج `confidence_score`.

**السلوك الواقعي المحتمل:**

- Headers البريد (From:, To:, Subject:) تُضاف للنص المُلصق
- النظام يحاول استخراج بيانات منها
- confidence_score = منخفض (40-60%)
- تُعرَض للمستخدم: "الثقة: 55% — هل تريد المتابعة؟"

**ماذا يفعل الموظف؟** يضغط "نعم" دون قراءة دقيقة. الضمان يُحفظ بـ raw_data فيه بيانات Headers بريد ضمن حقول مثل `supplier_name`.

### نتيجة سلامة البيانات

❌ Data integrity مُخترَقة — raw_data يحتوي بيانات غير صحيحة
✅ يُسجَّل في التاريخ مصدر "paste" مع confidence_score الأصلي
⚠️ لا يوجد أي تحقق لاحق يُدقق في التناسق بين confidence_score ومحتوى raw_data

## السيناريو الثالث — التصحيح اليدوي المتكرر ثم الاستيراد المكرر

### الوصف الواقعي

موظف يُصحح بيانات ضمان يدوياً (اسم مورد، تاريخ انتهاء). بعد أسبوع يستورد نفس ملف Excel المُحدَّث من قِبَل المورد. الملف يحتوي نفس رقم الضمان.

### ردّ فعل النظام

```php
// ImportService
if (ضمان موجود بنفس guarantee_number) {
    recordDuplicateImportEvent() // يُسجَّل كـ "استيراد مكرر"
    return; // لا تحديث
}
```

**المشكلة:** التصحيح اليدوي الذي أجراه الموظف **محفوظ**. لكن الضمان المُحدَّث في ملف Excel الجديد (والذي قد يحتوي تصحيحات من المورد) **مُتجاهَل تماماً بصمت**.

### السجل الجنائي

✅ يُسجَّل `duplicate_import` في التاريخ
⚠️ لكن لا يُوضَّح للمستخدم ماذا كانت الفروقات بين الاستيراد المكرر والبيانات الموجودة

### اتساق سير العمل

❌ خلل محتمل: الضمان موجود بحالة `audited` (جرى تدقيقه)، لكن البيانات الأساسية قد تكون مختلفة عن المستند الرسمي

## السيناريو الرابع — workflow مُتقدِّم ومُرجَع متكراراً

### الوصف الواقعي

مراجع يُقدِّم ضمان إلى مرحلة `audited`. مشرف يرى خطأً، يرفضه. يرجع لـ `draft`. يُصحَّح، يُتقدَّم. يُرفَض مجدداً. هذه الدورة تتكرر 5 مرات.

### ردّ فعل النظام

في كل `reject` → كل الحقول تُعاد لـ `draft`:

```php
workflow_step = 'draft'
signatures_received = 0
```

في كل `advance` تُنشأ:

- snapshot جديد في `guarantee_history`
- patch_data من الحالة السابقة
- `actor_display` للموظف

### السجل الجنائي

✅ `guarantee_history` يحتوي **10 أحداث** (5 advance + 5 reject) — التاريخ الكامل محفوظ
⚠️ **مشكلة تكلفة التخزين:** كل حلقة تُنتج 2 أحداث × snapshot/patch. ضمان يمر بـ 20 دورة رفض = 40 حدث. بعد آلاف الضمانات الإشكالية: `guarantee_history` ينتفخ

### اتساق سير العمل

✅ DB trigger يمنع أي تحول غير مشروع
⚠️ **خلل UX:** لا يوجد مؤشر لعدد مرات الرفض في الواجهة. مراجع لا يعرف أن هذا الضمان رُفض 5 مرات مسبقاً — قد يُصادق دون تمحيص إضافي

## السيناريو الخامس — عملية دفعة جماعية في نهاية اليوم المتسرعة

### الوصف الواقعي

مدير عمليات مضغوط وقت إغلاق الدوام. يختار دفعة بـ 200 ضمان. يُنفِّذ "تمديد جماعي" دون مراجعة كل ضمان. بعضها في مراحل workflow لم تكتمل.

### ردّ فعل النظام

`BatchService.extendBatch()` يُصنِّف كل ضمان:

- `ready` + غير مقفول → يُمتَّد ✅
- `pending` أو في workflow → `blocked: 'غير جاهز'` ⚠️
- مقفول → `blocked: 'مقفل'` ⚠️

النتيجة: `extended_count: 127, blocked: 73`

**ماذا يفعل الموظف؟** يُغلق النافذة دون قراءة الـ 73 المحجوبة. بعد 3 أيام تُكتشَف ضمانات منتهية الصلاحية لم تُمدَّد.

### سلامة البيانات

✅ النظام لم يُفسد بيانات
❌ **أثر تشغيلي حقيقي:** ضمانات منتهية الصلاحية دون تمديد = خسارة مالية محتملة

### نقطة التضخيم

لا يوجد في الواجهة: تنبيه تلقائي للمشرف إذا كانت نسبة المحجوبة > 30%. الـ `OperationalAlertService` لا يُراقب هذه الحالة.

## السيناريو السادس — سوء استخدام Break Glass

### الوصف الواقعي

مدير يريد الموافقة على ضمان لكن نظيره (المطلوب موافقته) في إجازة. بدلاً من انتظار العودة، يستخدم `break_glass` بسبب وهمي.

**الإدخال المقدَّم:**

- reason: "الزميل في إجازة وهناك موعد نهائي" (15 حرفاً ✓)
- ticket: "TK-000" (6 أحرف ✓ || **قد يكون وهمية**)
- TTL: 60 دقيقة

### ردّ فعل النظام

```php
// BreakGlassService
if (mb_strlen($reason) >= 8) ✓ // يمر
if (!empty($ticket)) ✓ // يمر (أي قيمة تمر)
```

**المشكلة:** النظام لا يتحقق من صحة رقم التذكرة في أي نظام خارجي. `BREAK_GLASS_REQUIRE_TICKET = true` يُلزم بوجود قيمة لكن **قيمة فارغة أو وهمية تمر**.

### السجل الجنائي

✅ `break_glass_events` يُسجَّل بالكامل مع السبب والتذكرة المُدخَلة
✅ إشعار يُرسَل لكل المديرين
⚠️ **لكن:** إذا لم ينتبه أحد للإشعار (مشغولون في نهاية الدوام)، الاستثناء مرَّ

### نقطة الاعتماد على سلوك المستخدم

🔴 النظام يعتمد كلياً على أن **شخصاً ما سيقرأ إشعار break glass**. لا يوجد escalation تلقائي إذا لم يُستجَب.

## السيناريو السابع — تكرار الاستيراد بسبب الارتباك

### الوصف الواقعي

موظف يستورد Excel. يُنتِج النظام نتيجة. لا يُلاحظ النتيجة. يستورد نفس الملف مجدداً.

### ردّ فعل النظام

**الضمان الأول (جديد):** يُضاف لـ DB
**الضمان الثاني (نفس `guarantee_number`):**

- `UNIQUE constraint` يمنع الإضافة
- `recordDuplicateImportEvent()` يُسجَّل
- يُرجَع نجاح جزئي للمستخدم بدون خطأ واضح

### سلامة البيانات

✅ تكامل البيانات محفوظ — لا تكرار
❌ **سلامة Workflow:** إذا كان الاستيراد الأول جزئياً (100 ضمان من 200)، ثم أُعيد استيراد الملف كاملاً → النظام يُضيف 100 الجديدة ويتجاهل 100 الموجودة بصمت. المستخدم يعتقد أن الـ 200 معالجة كلها.

### السجل الجنائي

✅ `guarantee_history` يسجل `duplicate_import` للضمانات الموجودة
⚠️ لكن لا يُوضح للمستخدم الفرق بين "مضاف" و"موجود مسبقاً"

## السيناريو الثامن — إدخال بيانات غير متسقة بين المستخدمين

### الوصف الواقعي

3 موظفين يدخلون ضمانات يدوياً لنفس المورد:

- الأول: `اسم المورد: Aramco`
- الثاني: `اسم المورد: Saudi Aramco`
- الثالث: `اسم المورد: أرامكو السعودية`

### ردّ فعل النظام

كل ضمان يمر بالمطابقة الذكية:

- `Aramco` → قد يُطابق `Saudi Aramco` بثقة 70%
- `Saudi Aramco` → بثقة أعلى
- `أرامكو السعودية` → لا تطابق (FuzzySignalFeeder معطوب للعربية)

**النتيجة:** نفس الكيان الحقيقي يظهر في النظام بثلاثة supplier_ids مختلفة في بعض الضمانات، مُرتبطاً بنفس supplier_id في أخرى.

### نقطة الاعتماد الخطيرة

🔴 لا يوجد **Supplier Deduplication UI** — لا طريقة للمستخدم ليكتشف أن `Aramco` و`أرامكو السعودية` ربطا بموردَين مختلفَين.
🔴 التقارير الجمالية ستُظهر كيانات مكررة — بيانات مالية خاطئة.

## السيناريو التاسع — طلب undo بعد فوات الأوان

### الوصف الواقعي

مدير يكتشف بعد أسبوعين من التمديد أن التمديد كان خاطئاً. يُقدِّم طلب undo.

### ردّ فعل النظام

`UndoRequestService.submit()`:

```php
if ($decision['status'] === 'released') {
    return error('لا يمكن التراجع عن ضمان مُفرَج عنه');
}
```

إذا لم يُفرَج بعد → طلب undo يُقدَّم → ينتظر موافقة شخص آخر (3-actor chain)

**أما إذا:** الضمان مَرَّ بـ 3 تمديدات بعد التمديد الخاطئ، كلها `committed`:

- undo يُعيد آخر تمديد فقط
- النظام لا يدعم undo متعدد المستويات (سلسلة)
- المستخدم يجب أن يُقدِّم طلبات undo متسلسلة

**السلوك الواقعي:**
الموظف لا يعرف عدد مستويات الـ undo المطلوبة. يُقدِّم طلباً واحداً، يرى أن الضمان لم يعد لحالته الأصلية، يشعر بالارتباك.

## السيناريو العاشر — سوء قراءة نتائج المطابقة الذكية

### الوصف الواقعي

موظف يُراجع قائمة مقترحات المورد. يرى:

```
الخيار 1: شركة الشرق للمقاولات (ثقة: 87%) [يوجد 3 ضمانات سابقة]
الخيار 2: شركة شرق الأوسط للمقاولات (ثقة: 72%)
```

يختار الخيار 1 حسب الثقة الأعلى، لكن الخيار الصحيح هو 2.

### ردّ فعل النظام

✅ يُضيف decision كـ `supplier_id = 1` (الخطأ)
✅ يُسجَّل في التاريخ: `decided_by = user_X`, `decision_source = 'manual'`
❌ **لا يوجد** تنبيه "هل أنت متأكد من الاختيار؟" قبل الحفظ

### نقطة التضخيم الأعلى خطورة

🔴 `UnifiedLearningAuthority` سيتعلم من هذا الاختيار اليدوي (إذا كان التعلم الإيجابي مُفعَّلاً):

- **يُعزِّز** الرابط الخاطئ `"شرق الأوسط" → شركة الشرق`
- يُصعِّب التصحيح مستقبلاً
- الضمانات التالية لنفس المورد ستحصل تلقائياً على اختيار أعلى ثقة — لكنه **خاطئ**

## السيناريو الحادي عشر — معالجة workflow بدون فهم المراحل

### الوصف الواقعي

موظف جديد أُسنِدَت إليه صلاحية `audit_data`. يرى أمامه ضمانات في مرحلة `draft`. يضغط "تقدم" على كل ضمان دون مراجعة البيانات.

### ردّ فعل النظام

✅ `WorkflowService.advance()` يتحقق من الصلاحية (موجودة)
✅ يُسجِّل التقدم بـ `actor_user_id`
❌ **لا يوجد** checklist إلزامي قبل التقدم
❌ **لا يوجد** minimum fields validation (هل raw_data مكتملة؟)

### نقطة الاعتماد على التدريب

🔴 **اعتماد كامل على التدريب:** النظام يثق بأن صاحب صلاحية `audit_data` يعرف ما يُدقِّق فيه. لا safeguard يمنع موظفاً غير مدرَّب من تمرير بيانات ناقصة.

**الأثر التراكمي:** ضمانات وصلت لمرحلة `analyzed` بعد مراجعة شكلية فارغة → quality degradation صامت في الـ workflow.

## السيناريو الثاني عشر — فتح نفس الضمان في تابين متعددين

### الوصف الواقعي

موظف يفتح ضماناً في تاب للإطلاع، يفتح نسخة ثانية لتعديل، يُحدِّث البيانات في التاب الثاني، ينتقل للأول ليُنفِّذ إجراءً.

### ردّ فعل النظام

التاب الأول يحمل state قديمة (snapshot بدون المعلومات المُحدَّثة). عند تنفيذ إجراء:

- **`createSnapshot()`** يأخذ snapshot **من DB الحالية** (ما غيَّره التاب الثاني)
- الإجراء يُنفَّذ على الحالة الصحيحة ✅

لكن:

- User يتوقع أن الإجراء يُطبَّق على ما رآه في الـ UI
- الـ UI في التاب الأول قديم → **disconnect بين ما يظنه المستخدم وما يحدث فعلاً**

### السجل الجنائي

⚠️ التاريخ يُسجِّل الحالة الفعلية (الصحيحة من DB)، لكن المستخدم قد يُفيد بأنه نفَّذ إجراءاً على حالة مختلفة — تناقض في شهادات الموظفين أمام الجهات الرقابية.

## السيناريو الثالث عشر — ضغط "رفض" بدلاً من "تقدم" عن طريق الخطأ

### الوصف الواقعي

مُشرف مُرهَق يعالج 30 ضماناً. يضغط "رفض" بدلاً من "تقدم" على ضمان مهم.

### ردّ فعل النظام

```
workflow_step → 'draft'
signatures_received → 0
```

التسجيل في `guarantee_history`:

```
event_type: 'workflow_rejection'
workflow_step_before: 'supervised'
workflow_step_after: 'draft'
actor_user_id: supervisor_id
```

**إمكانية الرجوع؟**

- يجب إعادة تمرير الضمان بجميع مراحل (draft→audited→analyzed→supervised) **من جديد**
- لا يوجد **"إلغاء آخر إجراء"** برمجي للـ workflow
- `UndoRequestService` يتعامل مع تمديد/تخفيض/إفراج — **لا يتعامل مع workflow rejections**

### نقطة التضخيم

🔴 خطأ ضغطة واحدة = إعادة workflow كاملة = 4 مراحل × 4 موظفين مختلفين = **ساعات من العمل الإضافي**. لا تراجع، لا undo.

## السيناريو الرابع عشر — إدخال مبالغ بصيغ مختلفة

### الوصف الواقعي

موظفون يدخلون المبالغ بصيغ متنوعة:

- `1500000`
- `1,500,000`
- `1.5M`
- `١٫٥ مليون`

### ردّ فعل النظام

`FieldExtractionService` يُحاول تطبيع المبالغ.

**السلوك الفعلي حسب المصدر:**

- Excel: يقرأ القيمة كـ float تلقائياً ✅
- Paste: `'1,500,000'` → `(float)'1,500,000'` = `1.0` في PHP! ❌
- Manual: يعتمد على validation الـ frontend (غير قابل للتحقق)
- `1.5M` → يُعاد كـ `1.5` في أحسن الأحوال

**المشكلة الحرجة:** `(float)'1,500,000'` في PHP = **1.0** (PHP يتوقف عند أول حرف غير رقمي). المبلغ يُحفظ كـ 1 ريال بدلاً من 1,500,000.

### سلامة البيانات

🔴 تلف صامت في المبلغ — يُحفظ `raw_data→amount = 1.0` بدون أي خطأ أو إنذار

### السجل الجنائي

✅ يُسجَّل `import_source = 'paste'` و confidence_score الأصلي
❌ لكن `raw_data.amount` محفوظ بقيمة خاطئة — السجل صحيح جنائياً لكن البيانات فاسدة

## السيناريو الخامس عشر — تغيير الإعدادات بدون فهم التتابعية

### الوصف الواقعي

مدير نظام يُقلِّل `MATCH_AUTO_THRESHOLD` من 90 إلى 70 "لتسريع المعالجة". لا يُدرك أن هذا يعني أن موردين بتشابه 70% سيحصلان على auto-match بدلاً من Review.

### ردّ فعل النظام

الإعداد يسري فوراً على كل استيراد جديد. في اليوم التالي:

- 300 ضمان تُعالَج آلياً كانت ستحتاج مراجعة بشرية
- بعضها بمطابقات فازية خاطئة تُقبَل تلقائياً

✅ `settings_audit_logs` يُسجِّل التغيير: من `0.90` إلى `0.70` باسم المدير

## السيناريو السادس عشر — تراكم طلبات Undo المعلقة

### الوضع الواقعي

موظفون يُقدِّمون طلبات undo لكن المعتمد في إجازة. 10 طلبات تتراكم.

**ردّ النظام:**
لا timeout تلقائي لطلبات الـ undo. تبقى `pending` إلى الأبد حتى يتحرك شخص.

**الأثر المتراكم:**
🔴 ضمانات "مُجمَّدة تشغيلياً" لأن `active_action` مضبوط ولا يمكن تمريرها على workflow آخر بدون إغلاق الـ undo
🔴 `OperationalAlertService` يراقب `pending undo requests` لكن فقط عند استدعاء الكود — **لا alerts proactive في وقت حقيقي**

## الخريطة التركيبية — نقاط تضخيم الخطأ البشري

```
المستخدم                    سلوك النظام              الأثر
─────────────────────────────────────────────────────────────────────
اختيار مورد خاطئ    ──→  تعلم الاختيار      ──→  ضمانات مستقبلية خاطئة آلياً
ضغط تقدم عشوائي   ──→  لا validation       ──→  workflow لا يعكس فحصاً حقيقياً
مبلغ بصيغة خاطئة  ──→  تحويل PHP صامت      ──→  بيانات مالية فاسدة
break_glass وهمي   ──→  يمر بتذكرة فارغة    ──→  تحايل غير مُلاحَظ
دفعة عشوائية      ──→  نجاح جزئي صامت      ──→  ضمانات مفاجئة منتهية
إعدادات متسرعة    ──→  تسري فوراً          ──→  auto-match لمطابقات خاطئة
تابات متعددة      ──→  state stale         ──→  تناقض بين ما فُعِل وما يُعتقَد
```

## جدول نقاط الاعتماد على السلوك الصحيح

| المجال                            | درجة الاعتماد | ماذا يحدث عند الخطأ؟  | يوجد safeguard؟ |
| --------------------------------- | ------------- | --------------------- | --------------- |
| اختيار supplier صحيح              | 🔴 عالية جداً | learning يتعلم الخطأ  | لا              |
| قراءة نتائج confidence            | 🟡 عالية      | قرار خاطئ محفوظ       | لا              |
| مراجعة فعلية في كل workflow stage | 🔴 عالية جداً | workflow شكلي         | لا              |
| إدخال مبلغ بالصيغة الصحيحة        | 🟡 متوسطة     | تلف مالي صامت         | في بعض المسارات |
| استخدام break_glass بمبرر حقيقي   | 🟡 متوسطة     | يمر بتذكرة وهمية      | جزئياً          |
| قراءة نتائج batch قبل الإغلاق     | 🟡 عالية      | ضمانات معلقة مجهولة   | لا              |
| فهم أثر تغيير الإعدادات           | 🔴 عالية جداً | تغيير سياسات المطابقة | لا              |
| تقديم undo في الوقت المناسب       | 🟡 متوسطة     | تراكم معلق إلى الأبد  | جزئياً          |

## الإجراءات التصحيحية الموصى بها

### أولوية قصوى (تُعالج أثراً بشرياً حقيقياً):

**1. Confirmation Dialog قبل Workflow Advance/Reject**

```
"أنت على وشك رفض هذا الضمان من مرحلة 'supervised' إلى 'draft'
لا يمكن التراجع عن هذا الإجراء بدون workflow كامل جديد
تأكيد الرفض؟ [نعم] [إلغاء]"
```

**2. تحذير عند تغيير إعدادات المطابقة**
`"هذا التغيير سيؤثر على سلوك المطابقة الآلية لجميع الاستيرادات القادمة"`

**3. Filter لصق بصيغة المبالغ**
التحقق من regex للمبلغ قبل حفظ `raw_data`:

```php
if (preg_match('/[\,\.المO]/u', $rawAmount)) {
    $amount = $this->cleanAmount($rawAmount); // parse فعلي
}
```

**4. Batch Result Acknowledgment**
إجبار المستخدم على تأكيد رؤية النتائج الكاملة قبل إغلاق نافذة الدفعة:
`"تم تمديد 127 ضمان. 73 ضمان لم يُعالَج. هل قرأت تفاصيل الضمانات المحجوبة؟"`

**5. Supplier Conflict Warning**
قبل حفظ اختيار manual: إذا كان اختيار المستخدم يختلف عن ما اقترحه النظام بثقة > 85%:
`"النظام مطمئن بنسبة 87% لهذا المورد. أنت تختار موردًا مختلفًا — هل أنت متأكد؟"`

**6. Break Glass Escalation Timeout**
إذا لم يُستجَب لإشعار break_glass خلال 30 دقيقة → إشعار تصعيد تلقائي.

## الخلاصة التشخيصية

> WBGL نظام مُصمَّم بافتراض ضمني: **المستخدمون مدرَّبون جيداً ويعملون في بيئة هادئة**.
>
> الواقع التشغيلي مختلف تماماً: ضغط وقت، ارتباك واجهة، قرارات متسرعة، وانقطاعات.
>
> **أخطر نقطة:** نظام التعلم الذاتي يتعلم من الأخطاء البشرية ويُعزِّزها — خطأ اختيار واحد يُنتج عشرات الاقتراحات الخاطئة المستقبلية.
>
> **الأثر التراكمي:** النظام يبدأ بدقة عالية ثم يتدهور تدريجياً بسبب تغذية learning بأخطاء بشرية غير مُصحَّحة.
