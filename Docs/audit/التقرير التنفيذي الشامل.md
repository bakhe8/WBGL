القسم الأول: النموذج التشغيلي الحقيقي للنظام
1.1 — البنية المعمارية الفعلية
النظام يعمل كـ**"خدمات PHP مسطَّحة مع طبقة نمط Repository خفية"** وليس إطار عمل كامل. لا يوجد ORM، لا حاوية dependency injection، لا event bus. كل شيء يُنفَّذ عبر:


api/*.php  ←  الطبقة الخارجية (HTTP Entry Points)    ↓app/Services/*.php  ←  منطق الأعمال (Service Layer)    ↓app/Repositories/*.php  ←  طبقة الوصول للبيانات    ↓PostgreSQL  ←  raw PDO فقط
حقيقة جوهرية: مصدر الحقيقة الوحيد لبيانات الضمانات هو عمود guarantees.raw_data من نوع JSON — وليس أعمدة منفصلة ومُطبَّعة. هذا القرار له أثر عميق على كل شيء في النظام.

1.2 — دورة حياة الضمان: المسار الكامل

1.4 — نظام التاريخ الهجين V2: التحليل العميق
النظام أكثر تطوراً مما يبدو. يعمل كـدفتر حساب هجين:

كل N حدث (N=10 افتراضياً): anchor كامل — لقطة JSON شاملة للحالة.
بين الأحداث: patch فقط (RFC 6902 تقريباً) — فرق التغيير فقط.
أحداث المعالم (import, release, manual_override): anchor إلزامي بغض النظر عن العدد.
هذا يتيح استرداد أي حالة تاريخية بكفاءة:
applyPatch(findNearestAnchor(eventId), patchChain) = الحالة في أي لحظة.

القسم الخامس: التقرير التنفيذي الشامل
5.1 — كيف يعمل نظام WBGL فعلياً
WBGL هو نظام إدارة خطابات ضمانات بنكية يعمل بـPHP صرف بدون إطار عمل. يستقبل بيانات الضمانات من ثلاثة مصادر (Excel، إدخال يدوي، لصق ذكي)، يُطابقها مع موردين وبنوك مُسجَّلين، يُمرِّرها عبر مسار عمل متعدد المراحل، ويُنشئ خطابات رسمية عند عمليات التمديد والتخفيض والإفراج.

مصدر الحقيقة الوحيد هو JSON. كل بيانات الضمان تعيش في guarantees.raw_data::jsonb بدون تطبيع في أعمدة منفصلة. هذا قرار معماري عميق الأثر: يُتيح مرونة استيراد فورية ويمنع schema migration للبيانات — لكنه يُعمي قاعدة البيانات عن أي constraint تحقق، ويُخفق في اكتشاف بيانات فاسدة حتى وقت الاستخدام.

نظام التعلم يعمل عبر ست مغذيات إشارات تُنتج درجات ثقة، يحسمها ConfidenceCalculatorV2 بصيغة موحَّدة (base score + confirmation boost - rejection penalty). عتبة الثقة 95% للمطابقة التلقائية عالية جداً بتصميم مقصود — معظم المطابقات تبقى تحت المراجعة البشرية.

نظام التاريخ الهجين V2 هو أكثر مكوِّنات النظام نضجاً. يجمع anchors كاملة مع patches JSON لتحقيق إعادة بناء جنائي دقيق لأي لحظة زمنية. إضافة لقطات HTML للخطابات في لحظة إنشائها يُشكِّل سجلاً قانونياً لا يمكن التلاعب به عبر واجهة النظام.

5.3 — نقاط الضعف البنيوية
الضعف الأول والأكثر خطورة: كاشف التكرار في PostgreSQL معطَّل منذ اليوم الأول. هذا ليس خللاً حديثاً — هو نتيجة نقل كود مكتوب لـSQLite إلى بيئة PostgreSQL دون تعديل سلسلة الخطأ. في نظام يستورد مئات الضمانات دفعةً، هذا يعني أن استيراد ملف يحتوي ضماناً موجوداً سابقاً يُفشل الاستيراد كله بدلاً من تجاوزه.
