طبيعة النظام
WBGL هو نظام إدارة ضمانات بنكية (Bank Guarantee Letters) مبني على PHP باستخدام PostgreSQL. يدير دورة حياة الضمانات من الاستيراد حتى الإفراج، عبر سير عمل متعدد المراحل مع صلاحيات دقيقة ونظام تدقيق متكامل.

API Layer	api/ (~50 ملفاً)	نقاط نهاية HTTP مباشرة بدون Framework
App Layer	app/Services/ (~50 خدمة)	منطق الأعمال المركزي
Repository Layer	app/Repositories/	وصول البيانات فقط
Support Layer	app/Support/	مرافق النظام (Auth, DB, Security…)
Views	views/, partials/, templates/	واجهة المستخدم
Migrations	database/migrations/ (28 ملف)	تطور مخطط قاعدة البيانات

[BUG-01] فشل تحديد التكرار عند الاستيراد من Excel — خطأ قاعدة البيانات خاطئ

هذا الملف يُعرِّف دالة mb_levenshtein وهي دالة موجودة في PHP مع امتدادات معينة. إذا كانت الامتداد مُثبَّتاً في الخادم، ستحدث Fatal Error: Cannot redeclare mb_levenshtein. يجب أن يكون الملف محاطاً بـ if (!function_exists('mb_levenshtein')).

[DEBT-03] ازدواجية في فحص CSRF: autoload.php و api/_bootstrap.php
autoload.php يُجري فحص CSRF لغير المسارات التي تحتوي /api/
api/_bootstrap.php يُجري فحص CSRF لجميع المسارات المتغيِّرة
المشكلة: هذا يعتمد على أن $_SERVER['REQUEST_URI'] يحتوي بالفعل على /api/ — وهو افتراض fragile إذا تغيير هيكل URL. لا يوجد توثيق واضح لهذا الاختيار.

[DEBT-04] Settings::get() يُشغّل loadFileData() في كل استدعاء

<?phppublic function get(string $key, mixed $default = null): mixed{    $all = $this->all();  // يُحمّل ملفين JSON في كل مرة!    return $all[$key] ?? $default;}
Settings::all() → loadPrimary() + loadFileData() — قراءة الملفات من القرص في كل استدعاء. وبما أن Settings::getInstance() يُستخدم في Database::connect(), SessionSecurity, SecurityHeaders, Logger, فهذا يعني عشرات قراءات الملف في كل طلب HTTP.

[QUAL-04] استخدام \Exception بدلاً من \Throwable في بعض catch blocks
المواقع:

api/workflow-advance.php:133 — catch (\Exception $e)
api/reopen.php:82 — catch (Exception $e)
في PHP 7+، يجب استخدام \Throwable لالتقاط كل الأخطاء. \Error (خطأ PHP داخلي) لا يُلتقط بـ \Exception. بعض حالات التعذر (مثل TypeError في strict mode) ستفلت.

[QUAL-05] database/migrations لا تحتوي على ملف rollback
كل ملف migration يحتوي على UP فقط. لا يوجد DOWN أو ملفات rollback. في حالة اكتشاف مشكلة بعد migration، الفريق مضطر لكتابة SQL يدوي للتراجع.
