# WBGL - الخطة الشاملة للتصحيح والتثبيت والحوكمة التشغيلية

**الإصدار:** 1.0  
**التاريخ:** 2026-03-05  
**النوع:** وثيقة تنفيذية هندسية ملزمة لفريق التطوير والتشغيل  
**الغرض:** توثيق الوضع الحالي بالكامل، المشاكل الجذرية، بدائل الحل، خارطة طريق التصحيح، والنتائج المتوقعة مع خطة اختبارات تفصيلية.

---

## 1) الملخص التنفيذي

نظام WBGL يعمل وظيفيًا ويخدم المستخدمين، لكنه مر بمراحل بناء تراكمية أدت إلى:

1. تداخل مفاهيمي بين `workflow_step` و`status` و`is_locked`.
2. تسرب بيانات اختبارية إلى السياق التشغيلي في فترات سابقة.
3. تفاوتات سلوكية بين الشاشات في الفلاتر/العدادات/الرؤية حسب الدور.
4. حاجة واضحة لتقوية الحوكمة التشغيلية عبر قواعد بياناتية (invariants) واختبارات آلية.

الهدف من هذه الخطة ليس إعادة كتابة النظام، بل **إعادة ضبطه تدريجيًا** ليصبح:

1. قابلًا للتدقيق العالي.
2. ثابتًا سلوكيًا عبر كل الواجهات.
3. واضحًا للمستخدم غير التقني.
4. منخفض المخاطر عند إضافة ميزات مستقبلية.

---

## 2) نطاق الوثيقة

تشمل هذه الخطة:

1. واجهة التشغيل الرئيسية (`index.php`) ومسارات التنقل والفلترة.
2. سياسات الرؤية والصلاحيات (`GuaranteeVisibilityService`, `NavigationService`, `Guard`).
3. دورة اعتماد الضمانات (`workflow_step`).
4. حالة الجاهزية التشغيلية (`status`) والقفل (`is_locked`).
5. عزل بيانات الاختبار عن البيانات الحقيقية.
6. اتساق العدادات والإحصائيات بين الصفحات.
7. جودة البيانات وقيود النزاهة.
8. اختبارات الوحدة والتكامل وGateات CI.

خارج النطاق في هذه النسخة:

1. إعادة تصميم UI/UX كاملة.
2. تفكيك monolith إلى microservices.
3. إعادة هيكلة جذرية لنموذج البيانات بدون توافق رجعي.

## 3) الحالة الحالية (Current State)

### 3.1 نموذج التشغيل الحالي

1. التسلسل الإداري للضمان:  
   `draft -> audited -> analyzed -> supervised -> approved -> signed`
2. **قاعدة بدء دورة الاعتماد المعتمدة في هذه الخطة:**
   1. السجل `ready` بدون `active_action` يبقى لدى `data_entry` فقط.
   2. لا يدخل السجل دورة الأدوار (`auditor -> analyst -> ...`) إلا بعد اختيار إجراء واضح:
      1. `extend`
      2. `reduce`
      3. `release`
3. المهام التنفيذية الفعلية (Actionable) للأدوار بعد `data_entry` تعتمد على:
   1. `status = 'ready'`
   2. `active_action` **غير فارغ** (`extend/reduce/release`)
   3. `is_locked = false`
   4. `workflow_step` ضمن صلاحيات الدور
4. بعد `signed` يعود السجل تشغيليًا إلى `data_entry` للطباعة والمتابعة.
5. التقدم بين المراحل يتم عبر `api/workflow-advance.php`.
6. الرؤية مقيدة حاليًا بنمط "مهام فقط" لكل الأدوار، مع استثناء:
   1. `data_entry`
   2. `developer`
   3. ومنح صلاحية استثنائية: `ui_full_filters_view`

### 3.2 ما تم تثبيته مؤخرًا

1. منع فتح السجلات خارج المهام للأدوار التشغيلية (task-only scope).
2. إخفاء الفلاتر الكاملة عن الأدوار التشغيلية افتراضيًا.
3. إضافة استثناء قابل للإدارة عبر Permission (`ui_full_filters_view`).
4. إبقاء السلوك الافتراضي مبنيًا على `role slug` مع إمكانية override.

### 3.3 الأعراض التي ظهرت ميدانيًا

1. عدم اتساق أعداد الضمانات/الدفعات بين صفحات مختلفة.
2. ظهور بيانات اختبارية على أنها حقيقية.
3. ظهور حسابات/أدوار تكاملية كثيرة في صفحة المستخدمين.
4. التباس للمستخدمين حول معنى:
   1. `draft`
   2. `pending`
   3. `ready`
   4. `signed`
5. ظهور مفاتيح ترجمة غير مكتملة في بعض أجزاء الواجهة.

## 4) نموذج المفاهيم الصحيح المطلوب اعتماده

## 4.1 الفصل الصريح بين المفاهيم

1. **`workflow_step`** = المرحلة الإدارية (Role-driven approval chain).
2. **`status`** = الجاهزية التشغيلية للبيانات (`pending` أو `ready`).
3. **`is_locked`** = قفل تنفيذي/حوكمي يمنع التعديل.
4. **`active_action`** = عملية تشغيلية قيد التنفيذ (مثل release/extend/reduce).

## 4.2 قواعد الاتساق (Invariants) المستهدفة

1. السجل لا يدخل دورة الاعتماد إلا إذا:
   1. `status = ready`
   2. `active_action` غير فارغ (`extend/reduce/release`)
2. الأدوار بعد `data_entry` تتعامل تشغيليًا فقط مع سجلات:
   1. `ready`
   2. لها `active_action` محدد مسبقًا
3. السجل `ready` بدون `active_action` لا يغادر نطاق `data_entry`.
4. بعد الوصول إلى `signed` يعود السجل إلى `data_entry` لغرض الطباعة والمتابعة فقط.
5. الرفض من أي دور اعتماد يعيد السجل إلى:
   1. `workflow_step = draft`
   2. `status = ready`
   3. `active_action = null`
   4. مع ملاحظة رفض إلزامية
6. لا يسمح بخلط:
   1. ضمان اختباري داخل دفعة حقيقية
   2. ضمان حقيقي داخل دفعة اختبارية
7. لا يوجد ضمان دون انتماء دفعة (إلا حالات انتقالية محددة ومراقبة).
8. أي سجل يخالف invariants يدخل تقرير سلامة بيانات يومي.
9. كل حدث تشغيلي/حوكمي يجب أن يسجل هوية منفذ واضحة (اسم أو بريد أو معرف).
10. لا يُقبل حفظ أحداث/ملاحظات/مرفقات بهوية عرض عامة مثل `المستخدم` فقط.

## 5) قائمة المشاكل الجذرية (Root Problems)

## 5.1 تداخل مفهومي Stage/Status/Lock

**الوصف:** استخدام هذه المفاهيم بشكل غير موحّد في بعض الشاشات والتقارير أدى لقراءة خاطئة للحالة الفعلية.

**الأثر:**

1. قرارات تشغيلية خاطئة.
2. ضعف ثقة المستخدم في مصداقية الأرقام.

**السبب الجذري:**

1. نمو تدريجي للمنظومة مع تغييرات متتابعة.
2. عدم وجود قاموس مفاهيم رسمي ملزم ومترجم إلى قيود بياناتية.

---

## 5.2 تسرب بيانات الاختبار

**الوصف:** إدخالات اختبارية (خصوصًا تكاملية) ظهرت في التشغيل الفعلي عند عدم وسمها أو عدم عزلها جيدًا.

**الأثر:**

1. تشويه العدادات.
2. ضبابية في التقارير والإحصائيات.
3. مخاطر حوكمة.

**السبب الجذري:**

1. الاعتماد على وسم اختباري غير صارم في بعض المسارات.
2. عدم وجود قواعد نزاهة تمنع الخلط على مستوى DB.

---

## 5.3 عدم اتساق العدادات

**الوصف:** بعض الصفحات تعرض أرقامًا لا تتطابق بسبب اختلاف شروط الفلترة/الرؤية.

**الأثر:**

1. تناقض إداري أمام المالك.
2. صعوبة التشخيص عند الحوادث.

**السبب الجذري:**

1. أكثر من مصدر حساب للأرقام.
2. اختلاف السياق (task-only مقابل full scope) بدون إظهار السبب بوضوح.

---

## 5.4 تضخم مستخدمي التكامل/الاختبار

**الوصف:** وجود عدد كبير من مستخدمين بأسماء تكاملية أدى لتشويش صفحة المستخدمين.

**الأثر:**

1. صعوبة الإدارة اليومية.
2. احتمال منح صلاحيات غير دقيقة.

**السبب الجذري:**

1. اختبارات تكامل تنشئ مستخدمين متعددين.
2. غياب سياسة تنظيف دورية تلقائية/مشروطة.

---

## 5.5 ثغرات ترجمة وواجهة

**الوصف:** ظهور مفاتيح i18n بدل النص النهائي في بعض المكونات.

**الأثر:**

1. انطباع بعدم نضج المنتج.
2. سوء فهم للنصوص الحساسة.

**السبب الجذري:**

1. مفاتيح مفقودة أو fallback غير صارم.
2. تغييرات سريعة دون coverage i18n test.

---

## 5.6 غياب Policy موحدة مكتوبة ومفروضة

**الوصف:** السياسات موجودة فعليًا في الكود، لكن يلزم توحيدها في وثيقة + اختبارات + قيود DB.

**الأثر:**

1. اختلاف تفسير بين التطوير والتشغيل.
2. تكرار الجدال حول "المفترض".

**السبب الجذري:**

1. وثائق متفرقة وغير مُزامنة كفاية مع الكود.

---

## 5.7 ضعف إسناد هوية المنفذ في السجلات

**الوصف:** جزء من الأحداث يظهر بوسم عام مثل `المستخدم` بدل هوية منفذ فعلية يمكن تتبعها.

**الأثر:**

1. ضعف الشفافية والمساءلة التشغيلية.
2. صعوبة التحقيق عند الاعتراضات أو المراجعات.
3. انخفاض القيمة التدقيقية للـ Timeline والملاحظات والمرفقات.

**السبب الجذري:**

1. استخدام `display fallback` عام عند غياب الحقول.
2. عدم توحيد عقد هوية الفاعل (Actor Contract) عبر جميع المسارات.

**المزايا:**

1. سرعة عالية.

**المزايا:**

**الحكم:** غير مناسب حاليًا.

**الوصف:** برنامج تصحيح على مراحل:  
Policy موحدة + قيود DB + اختبارات + تنظيف بيانات + توحيد عدادات.

**المزايا:**

1. أقل مخاطرة.
2. يحافظ على الاستمرارية.
3. ينتج استقرارًا قابلًا للقياس.

**العيوب:**

1. يتطلب انضباط تنفيذي ومتابعة دقيقة.

**الحكم:** الخيار المعتمد.

## 7) التصميم المستهدف (Target Model)

## 7.1 قواعد التشغيل المعتمدة

1. **Data Entry**:
   1. يرى النطاق الواسع (حسب السياسة).
   2. مسؤول عن تحويل `pending -> ready`.
   3. هو **نقطة بدء الدورة** عبر تحديد `active_action`:
      1. `extend`
      2. `reduce`
      3. `release`
2. الأدوار بعده:
   1. تعمل على `Actionable` فقط.
   2. لا تتعامل تشغيليًا إلا مع `status=ready` + `active_action` محدد.
3. `workflow_step` يمثل الاعتماد الإداري فقط.
4. `status` لا يعبّر عن "منصة الاعتماد" بل "جاهزية البيانات".
5. `is_locked` يتقدم على أي محاولة تعديل.
6. `ready` بدون إجراء ليس عملًا للأدوار اللاحقة، ولا يذهب تلقائيًا للمدقق.
7. بعد `signed` يعود السجل إلى `data_entry` ويظهر له زر الطباعة حسب السياسة.
8. بعد الطباعة، **يبقى السجل لدى `data_entry`** كضمان موجود وجاهز لإجراء جديد لاحق دون إعادة إدخال.
9. الفرق تشغيليًا إلزامي بين:
   1. إدخال ضمان جديد: يبدأ من `draft/pending`.
   2. ضمان موجود جاهز: يبقى `ready` ويبدأ منه إجراء جديد مباشر.
10. الاستثناء الوحيد على مدخل البيانات:
   1. الضمان **المفرج عنه والموقّع** يعتبر حالة نهائية.
   2. لا يسمح عليه تعديل بيانات أو بدء إجراء جديد.
   3. يسمح بالعرض/الأرشفة فقط.
11. زر **تعديل بيانات النظام** يكون متاحًا لـ `data_entry` فقط.
12. الأدوار اللاحقة يظهر لها بدلًا منه زر **رفض**.
13. الرفض لا يحذف السجل، بل يعيده لـ `data_entry` بحالة `ready` وبدون `active_action`.
14. الرفض يتطلب ملاحظة إلزامية (سبب الرفض) تُحفظ في `notes`.
15. الاستثناء في الفلاتر الكاملة لا يُمنح إلا بقرار حوكمي عبر `ui_full_filters_view`.
16. منطق الدفعات تشغيليًا (تنفيذ إجراءات الدفعة وإدارة فتح/إغلاق الدفعات) محصور على `data_entry` افتراضيًا.
17. الأدوار اللاحقة (`data_auditor`, `analyst`, `supervisor`, `approver`, `signatory`) لا تستخدم الدفعات كمسار عمل يومي.
18. أي فتح لمنطق الدفعات خارج `data_entry` يتم فقط باستثناء حوكمي مستقل قابل للتدقيق (مثل `batch_full_operations_override`).
19. كل مسار في دورة حياة الضمان (استيراد، حفظ، انتقال مرحلة، رفض، توقيع، إعادة فتح، ملاحظات، مرفقات) يسجل Actor واضح.
20. Actor الأدنى المقبول للعرض: `display_name` أو `username` أو `email` أو `user_id`.
21. يمنع في الواجهة عرض قيمة عامة مثل `المستخدم` عندما تتوفر هوية فعلية.

## 7.2 سياسة الاستثناءات

1. الاستثناء لا يلغي التتبع.
2. أي استثناء يجب أن يكون:
   1. مبرر
   2. مؤقت عند الحاجة
   3. قابل للمراجعة من سجل الصلاحيات.

## 7.3 دليل الأدوار والصلاحيات والإجراءات (مرجع تشغيلي ملزم)

> هذا القسم يربط مباشرة بين الدور، الصلاحية، الإجراء المطلوب، وتأثيره على دورة حياة الضمان.

## 7.3.1 الأدوار الرسمية في النظام

1. `developer` (مطور/حوكمة تقنية)
2. `data_entry` (مدخل بيانات)
3. `data_auditor` (مدقق بيانات)
4. `analyst` (محلل)
5. `supervisor` (مشرف)
6. `approver` (مدير معتمد)
7. `signatory` (المفوض بالتوقيع)

## 7.3.2 مصفوفة الدور مقابل نطاق الرؤية

1. `developer`:
   1. نطاق واسع.
   2. الفلاتر الكاملة ظاهرة.
2. `data_entry`:
   1. نطاق واسع تشغيليًا.
   2. الفلاتر الكاملة ظاهرة.
   3. يرى ويدير صفحات الدفعات ضمن الصلاحيات التشغيلية.
3. الأدوار التشغيلية (`data_auditor`, `analyst`, `supervisor`, `approver`, `signatory`):
   1. افتراضيًا: نطاق مهام فقط (`Actionable`).
   2. لا يفتحون سجلات خارج المهام.
   3. الفلاتر الكاملة مخفية.
   4. لا يرون صفحات الدفعات افتراضيًا ولا ينفذون إجراءات الدفعات.
4. استثناء اختياري: `ui_full_filters_view`
   1. إذا مُنحت لدور أو مستخدم، تسمح بعرض الفلاتر الكاملة.
   2. الاستثناء خاضع للمراجعة الحوكمية.
5. استثناء دفعات حوكمي: `batch_full_operations_override`
   1. يسمح بتفعيل منطق الدفعات خارج `data_entry` عند حالات خاصة.
   2. لا يمنح دائمًا بدون تذكرة/مبرر وتدقيق.

## 7.3.3 مصفوفة المرحلة الإدارية مقابل الصلاحية

1. `draft -> audited`
   1. الصلاحية: `audit_data`
   2. الدور المستهدف: `data_auditor`
2. `audited -> analyzed`
   1. الصلاحية: `analyze_guarantee`
   2. الدور المستهدف: `analyst`
3. `analyzed -> supervised`
   1. الصلاحية: `supervise_analysis`
   2. الدور المستهدف: `supervisor`
4. `supervised -> approved`
   1. الصلاحية: `approve_decision`
   2. الدور المستهدف: `approver`
5. `approved -> signed`
   1. الصلاحية: `sign_letters`
   2. الدور المستهدف: `signatory`

## 7.3.4 شروط قابلية التنفيذ (Actionable) الموحدة

لا يعتبر السجل قابلًا للتنفيذ إلا حسب نوع الدور:

1. **`data_entry` (قبل بدء الدورة)**:
   1. `status = ready`
   2. `active_action` فارغ
   3. `workflow_step = draft`
   4. الهدف: اختيار الإجراء (`extend/reduce/release`)
2. **الأدوار اللاحقة (`data_auditor` حتى `signatory`)**:
   1. `status = ready`
   2. `active_action` غير فارغ
   3. `is_locked = false`
   4. `workflow_step` ضمن المراحل المسموحة للدور
3. **`data_entry` (بعد نهاية الدورة)**:
   1. `workflow_step = signed`
   2. `status = ready`
   3. الغرض: الطباعة والمتابعة
   4. غير مسموح إذا كان السجل نهائيًا: `workflow_step = signed` + `status = released`
4. في كل الحالات: السجل يجب أن يكون ضمن نطاق الرؤية policy للمستخدم.

## 7.3.5 إجراءات كل دور داخل الواجهة

### A) `data_entry` (مدخل البيانات)

1. الهدف: تحويل السجل من غير مكتمل إلى جاهز (`pending -> ready`).
2. الإجراءات:
   1. إدخال/تصحيح المورد.
   2. إدخال/تصحيح البنك.
   3. حفظ السجل.
   4. التحقق من الجاهزية (`ready`).
   5. اختيار الإجراء التشغيلي (`extend/reduce/release`) لبدء دورة الاعتماد.
3. نمطا العمل لدى `data_entry`:
   1. **نمط الإدخال الجديد (New Intake):**
      1. السجل جديد.
      2. يبدأ `draft/pending`.
      3. يتطلب إدخال المورد/البنك حتى يصبح `ready`.
   2. **نمط الإجراء اللاحق (Existing Ready):**
      1. السجل موجود أصلًا في النظام.
      2. عاد بعد `signed` أو `reject`.
      3. لا يحتاج إعادة إدخال من الصفر.
      4. يختار المدخل إجراء جديد مباشر (`extend/reduce/release`) ويبدأ دورة جديدة.
   3. **نمط نهائي مغلق (Released + Signed):**
      1. السجل مفرج عنه وموقّع.
      2. لا يقبل تعديل أو إجراء جديد.
      3. يبقى للعرض/الأرشفة فقط.
4. ما لا يفعله افتراضيًا:
   1. لا يدفع السجل يدويًا لمستخدم معين.
   2. لا يتجاوز سلسلة الاعتماد.
   3. لا يرسل سجلًا `ready` بدون `active_action` إلى المدقق.

### B) `data_auditor` (المدقق)

1. الهدف: تنفيذ خطوة التدقيق.
2. المدخل المتوقع: سجلات `ready` في مرحلة `draft` **بعد تحديد** `active_action`.
3. الإجراء:
   1. مراجعة السجل.
   2. الضغط على زر التقدم.
   3. أو الضغط على زر الرفض عند الحاجة.
   4. عند الرفض: إدخال سبب إلزامي في نافذة منبثقة.
4. المخرج: انتقال السجل إلى `audited`.
5. مخرج الرفض: إعادة السجل إلى `data_entry` كـ `ready` و`active_action=null` مع حفظ الملاحظة.

### C) `analyst` (المحلل)

1. الهدف: تنفيذ التحليل.
2. المدخل المتوقع: سجلات `ready` في مرحلة `audited` مع `active_action` معروف.
3. الإجراء:
   1. تحليل السجل.
   2. اعتماد التحليل عبر زر التقدم.
   3. أو رفض السجل مع سبب إلزامي.
4. المخرج: انتقال السجل إلى `analyzed`.
5. مخرج الرفض: إعادة السجل إلى `data_entry` مع تصفير `active_action`.

### D) `supervisor` (المشرف)

1. الهدف: مراجعة التحليل.
2. المدخل المتوقع: سجلات `ready` في مرحلة `analyzed` مع نفس الإجراء المحدد.
3. الإجراء:
   1. مراجعة قرار التحليل.
   2. تمرير المرحلة عبر زر التقدم.
   3. أو رفض السجل مع سبب إلزامي.
4. المخرج: انتقال السجل إلى `supervised`.
5. مخرج الرفض: إعادة السجل إلى `data_entry` مع ملاحظة سبب الرفض.

### E) `approver` (المدير المعتمد)

1. الهدف: الاعتماد النهائي قبل التوقيع.
2. المدخل المتوقع: سجلات `ready` في مرحلة `supervised` مع إجراء واضح.
3. الإجراء:
   1. اعتماد القرار.
   2. تمرير المرحلة.
   3. أو رفض السجل مع سبب إلزامي.
4. المخرج: انتقال السجل إلى `approved`.
5. مخرج الرفض: إعادة السجل إلى `data_entry` كـ `ready` بدون إجراء.

### F) `signatory` (المفوض بالتوقيع)

1. الهدف: التوقيع وإكمال سلسلة الاعتماد.
2. المدخل المتوقع: سجلات `ready` في مرحلة `approved`.
3. الإجراء:
   1. تنفيذ التوقيع عبر زر التقدم.
   2. أو رفض السجل مع سبب إلزامي قبل التوقيع.
4. المخرج: انتقال السجل إلى `signed`.
5. بعد ذلك: يعود السجل تشغيليًا إلى نطاق `data_entry` للطباعة والمتابعة.
6. لا تنتهي دورة التشغيل عمليًا إلا بعد عودة السجل لمدخل البيانات وطباعة الخطاب.
7. مخرج الرفض: إعادة السجل إلى `data_entry` مع إزالة `active_action`.

### G) `developer` (حوكمة تقنية/تشغيل إداري)

1. الهدف: إدارة الصلاحيات، الاستثناءات، والصيانة التشغيلية.
2. الإجراءات:
   1. إدارة المستخدمين والأدوار.
   2. إدارة overrides.
   3. استخدام أدوات الصيانة والحوكمة.
   4. إدارة الاستثناءات مثل `ui_full_filters_view` و`batch_full_operations_override`.
   5. لا يستخدم مسار الدفعات تشغيليًا بشكل افتراضي، إلا عند تفعيل استثناء حوكمي موثق.

## 7.3.6 التداخل مع دورة حياة الضمان البنكي

دورة الاعتماد الإدارية:

1. `draft`
2. `audited`
3. `analyzed`
4. `supervised`
5. `approved`
6. `signed`

دورة الجاهزية التشغيلية:

1. `pending`: بيانات ناقصة، لا تدخل سلسلة تنفيذ الأدوار.
2. `ready`: بيانات مكتملة، قابلة للتنفيذ حسب المرحلة والصلاحية.

دورة الإجراء والتوجيه:

1. `ready` بدون `active_action`:
   1. يبقى عند `data_entry`.
   2. لا يذهب لأي دور اعتماد.
2. عند اختيار `extend/reduce/release`:
   1. يبدأ مسار الاعتماد كاملًا من `draft -> ... -> signed`.
3. بعد `signed`:
   1. يعود السجل إلى `data_entry` للطباعة.
   2. بعد الطباعة يبقى السجل كـ `ready` (بدون `active_action`) بانتظار إجراء لاحق عند الحاجة.
4. عند الحاجة لإجراء جديد على نفس الضمان:
   1. لا يتم إنشاء سجل جديد.
   2. لا تعاد دورة الإدخال من الصفر.
   3. يحدد `data_entry` إجراءً جديدًا على السجل الموجود ويبدأ مسار الاعتماد مجددًا.
5. إذا وصل السجل إلى حالة `released + signed`:
   1. ينتهي المسار التشغيلي لهذا الضمان.
   2. يمنع بدء أي `active_action` جديد عليه.
6. `is_locked=true` يمنع التعديل التشغيلي.

## 7.3.7 قواعد المنع الصريحة (Mandatory Deny Rules)

1. لا انتقال مرحلة إذا `status != ready`.
2. لا انتقال مرحلة إذا السجل مقفل.
3. لا انتقال مرحلة في سلسلة الاعتماد إذا `active_action` فارغ.
4. لا فتح سجل خارج نطاق الرؤية policy.
5. لا عرض فلاتر كاملة للأدوار التشغيلية إلا باستثناء صريح.
6. لا تحويل تلقائي لسجل `ready` إلى المدقق بدون اختيار إجراء.
7. لا يسمح بتنفيذ `reject` بدون إدخال سبب رفض كنص محفوظ في `notes`.
8. لا يسمح لـ `data_entry` بتعديل أو إضافة إجراء جديد إذا كان السجل `released + signed`.
9. لا يسمح لأي دور غير `data_entry` بتنفيذ إجراءات الدفعات (`extend/reduce/release/close/reopen/update_metadata`) دون استثناء حوكمي صريح.

## 7.3.8 سياسة الصلاحيات: دور أم مستخدم؟

الترتيب المعتمد:

1. الدور (`role`) هو المصدر الأساسي للصلاحيات.
2. التخصيص الفردي (`user override`) يستخدم كاستثناء فقط.
3. `deny` الفردي يعلو على `allow`.
4. أي استثناء يجب توثيقه وتدقيقه دوريًا.
5. استثناء الدفعات (`batch_full_operations_override`) لا يُمنح إلا لحالة حوكمة موثقة ومحددة زمنيًا.

## 7.3.9 مصفوفة تشغيل يومية لكل دور

1. مدخل البيانات:
   1. يراجع `pending`.
   2. يحوله إلى `ready`.
   3. يحدد `active_action` (extend/reduce/release) لبدء الدورة.
   4. يستلم `signed` للطباعة بعد اكتمال التوقيع.
   5. بعد الطباعة يبقي السجل متاحًا لديه كـ `ready` لإجراء لاحق دون إعادة إدخال.
   6. إذا كان السجل `released + signed` يتوقف عنده كحالة نهائية للعرض فقط.
   7. يدير الدفعات (فتح/إغلاق/تعديل/تنفيذ جماعي) كقناة تشغيل خاصة به.
2. المدقق:
   1. يعمل على `draft + ready + active_action`.
   2. ينقله إلى `audited`.
3. المحلل:
   1. يعمل على `audited + ready + active_action`.
   2. ينقله إلى `analyzed`.
4. المشرف:
   1. يعمل على `analyzed + ready + active_action`.
   2. ينقله إلى `supervised`.
5. المعتمد:
   1. يعمل على `supervised + ready + active_action`.
   2. ينقله إلى `approved`.
6. المفوض:
   1. يعمل على `approved + ready + active_action`.
   2. ينقله إلى `signed`.
7. جميع الأدوار بعد `data_entry`:
   1. لا تعتمد على شاشة الدفعات لتنفيذ دورها.
   2. تعتمد فقط على قائمة المهام المرحلية (`Actionable`).

## 7.3.10 اختبارات قبول خاصة بالأدوار

1. عند تسجيل الدخول بكل دور تشغيلي:
   1. يرى فقط سجلات مهامه القابلة للتنفيذ.
   2. لا يستطيع فتح سجل خارج المهام بالرابط المباشر.
2. عند منح `ui_full_filters_view` لمستخدم تشغيلي:
   1. تظهر له الفلاتر الكاملة.
   2. التتبع الأمني يسجل منحه/سحبه.
3. عند سحب `ui_full_filters_view`:
   1. يعود مباشرة إلى task-only scope.

## 7.3.11 سياسة الرفض وإرجاع السجل (ملزمة)

1. على مستوى الواجهة:
   1. زر تعديل بيانات النظام يظهر لـ `data_entry` فقط.
   2. بقية الأدوار يظهر لها زر `رفض` بدلًا من زر التعديل.
2. على مستوى السلوك:
   1. ضغط زر `رفض` لا يحذف الضمان.
   2. يعيد الضمان إلى `data_entry`.
   3. يضبط السجل إلى:
      1. `workflow_step = draft`
      2. `status = ready`
      3. `active_action = null`
3. على مستوى الحوكمة:
   1. الرفض بدون ملاحظة ممنوع.
   2. تظهر نافذة منبثقة لإدخال سبب الرفض (إلزامي).
   3. السبب يُحفظ كـ `note` مرتبطة بالضمان.
   4. مدخل البيانات يقرأ السبب من قسم الملاحظات عند رجوع السجل.
4. على مستوى التدقيق:
   1. تسجيل حدث timeline صريح لعملية الرفض.
   2. تسجيل هوية الرافض ووقت الرفض ونص السبب.

## 7.3.12 سياسة هوية المنفذ الموحدة (Actor Attribution)

1. نطاق الإلزام:
   1. Timeline events.
   2. Notes (إنشاء/تعديل).
   3. Attachments (رفع/حذف/تعديل).
   4. Lifecycle actions (import/save/advance/reject/reopen/sign/release/extend/reduce).
2. الحقول القياسية المطلوب تسجيلها لكل حدث:
   1. `actor_user_id` (إن وجد).
   2. `actor_username` (إن وجد).
   3. `actor_email` (إن وجد).
   4. `actor_display_name` (قيمة العرض الأساسية).
3. قاعدة العرض في الواجهة:
   1. أولوية العرض: `actor_display_name` ثم `actor_username` ثم `actor_email` ثم `actor_user_id`.
   2. لا يتم عرض نص عام `المستخدم` إلا عند غياب الهوية بالكامل في بيانات قديمة فقط.
4. التوافق الرجعي:
   1. البيانات التاريخية القديمة تبقى كما هي.
   2. أي حدث جديد بعد التفعيل يجب أن يحمل Actor واضحًا.
5. الحوكمة:
   1. أي endpoint لا يمرر هوية منفذ معتبرة يفشل validation.
   2. توحيد دالة استرجاع هوية المنفذ واستخدامها في جميع APIs.

## 8) خارطة الطريق التنفيذية (Roadmap)

> كل مرحلة لها مخرجات، اختبارات قبول، وخطة تراجع.

## المرحلة 0: تثبيت التعاريف (Concept Freeze)

**الأهداف:**

1. اعتماد القاموس الرسمي:
   1. `workflow_step`
   2. `status`
   3. `is_locked`
   4. `active_action`
2. توحيد المصطلحات في الوثائق الأساسية.

**المخرجات:**

1. تحديث SOP.
2. تحديث دليل الحوكمة التشغيلي.

**اختبارات القبول:**

1. لا يوجد تضارب تعريفات بين الوثائق.

---

## المرحلة 1: فرض Policy في كل مسار قراءة/تنقل

**الأهداف:**

1. منع أي bypass للرؤية خارج النطاق.
2. ضمان task-only للأدوار التشغيلية.

**المخرجات:**

1. توحيد تطبيق `GuaranteeVisibilityService`.
2. توحيد تطبيق `NavigationService`.
3. منع فتح سجل خارج المهام في كل APIs القراءة/التحريك.

**اختبارات القبول:**

1. مستخدم مدقق يرى مهامه فقط.
2. محاولة فتح `id` خارج النطاق => 403 أو empty out-of-scope.

---

## المرحلة 1.5: تفعيل مسار الرفض المعتمد (Reject Loop)

**الأهداف:**

1. جعل تعديل بيانات النظام حكرًا على `data_entry`.
2. توفير زر `رفض` للأدوار اللاحقة بدل زر التعديل.
3. إعادة السجل إلى `data_entry` بدون إجراء مع سبب رفض إلزامي.

**المخرجات:**

1. إظهار زر التعديل لـ `data_entry` فقط.
2. إظهار زر `رفض` لـ `data_auditor/analyst/supervisor/approver/signatory`.
3. نافذة منبثقة لسبب الرفض (حقل نصي إجباري).
4. عند الرفض:
   1. `workflow_step = draft`
   2. `status = ready`
   3. `active_action = null`
   4. حفظ السبب كـ `note`
   5. تسجيل حدث timeline للرفض

**اختبارات القبول:**

1. المستخدم غير `data_entry` لا يرى زر التعديل.
2. المستخدم غير `data_entry` يرى زر رفض فقط.
3. لا يمكن إرسال الرفض بدون سبب.
4. بعد الرفض يعود السجل إلى قائمة `data_entry` كـ `ready` بدون `active_action`.
5. مدخل البيانات يرى سبب الرفض ضمن الملاحظات.

---

## المرحلة 1.6: حصر منطق الدفعات على مدخل البيانات

**الأهداف:**

1. منع استخدام الدفعات كمسار عمل تشغيلي للأدوار اللاحقة.
2. تثبيت أن الدفعات قناة تشغيل لـ `data_entry` فقط افتراضيًا.
3. ضبط أي استثناء للدفعات عبر Permission مستقل قابل للتدقيق.

**المخرجات:**

1. تقييد صفحات الدفعات (`batches.php`, `batch-detail.php`, `batch-print.php`) بحيث تظهر فقط لـ `data_entry` افتراضيًا.
2. إضافة صلاحية استثناء حوكمي مخصصة (`batch_full_operations_override`) بدل التوسع عبر `manage_data`.
3. تقييد API الدفعات (`api/batches.php`) على صلاحية دفعات مخصصة، وليس `manage_data` العامة.
4. إخفاء أزرار إجراءات الدفعات عن غير المخولين.
5. تسجيل أي محاولة وصول أو تنفيذ مرفوضة كأثر تدقيقي.

**اختبارات القبول:**

1. `data_entry` يستطيع الوصول الكامل للدفعات وتنفيذ إجراءاتها ضمن السياسات.
2. `data_auditor/analyst/supervisor/approver/signatory`:
   1. لا يرون صفحات الدفعات.
   2. لا يستطيعون تنفيذ API الدفعات.
3. `developer` لا يعمل على الدفعات تشغيليًا إلا عند منح `batch_full_operations_override`.
4. أي وصول غير مخول للدفعات يرجع `403` مع تسجيل audit event.

---

## المرحلة 1.7: توحيد هوية المنفذ في كل السجلات (Actor Unification)

**الأهداف:**

1. إنهاء ظهور النص العام `المستخدم` في السجلات الجديدة.
2. توحيد مصدر هوية المنفذ عبر كل APIs والخدمات.
3. رفع الشفافية والتتبع في Timeline/Notes/Attachments.

**المخرجات:**

1. عقد موحد لهوية المنفذ (`actor_user_id/actor_username/actor_email/actor_display_name`).
2. تحديث كل مسارات lifecycle لتسجيل Actor وفق العقد الموحد.
3. تحديث واجهات العرض لقراءة actor fields بدل النص العام.
4. إضافة فحص يمنع إنشاء أحداث جديدة بدون Actor صالح.

**اختبارات القبول:**

1. أي حدث جديد في Timeline يظهر اسم/هوية منفذ محددة.
2. أي ملاحظة أو مرفق جديد يحمل هوية منفذ واضحة.
3. لا يظهر النص `المستخدم` في الأحداث الجديدة بعد التفعيل.
4. محاولة تنفيذ endpoint بدون actor context تُرفض.

---

## المرحلة 2: نزاهة البيانات (Data Integrity Hardening)

**الأهداف:**

1. منع خلط بيانات اختبار/حقيقي.
2. منع تناقض الضمانات/الدفعات.

**المخرجات:**

1. قيود DB تمنع:
   1. test guarantee داخل real batch
   2. real guarantee داخل test batch
2. تدقيق يومي تلقائي للتناقضات.
3. تقرير نزاهة دوري.

**اختبارات القبول:**

1. فشل أي عملية إدخال تخرق القيود.
2. تقرير النزاهة = صفر تناقضات.

---

## المرحلة 3: توحيد العدادات والإحصائيات

**الأهداف:**

1. مصدر حساب واحد للأرقام.
2. اتساق كامل بين:
   1. الصفحة الرئيسية
   2. الدفعات
   3. الصيانة
   4. الإحصائيات

**المخرجات:**

1. اعتماد service موحد للعدادات مع نفس predicates.
2. شرح UI للسياق (task-only vs full) حيث يلزم.

**اختبارات القبول:**

1. عدم وجود فرق غير مبرر بين الصفحات.
2. توثيق أي فرق مقصود كسياق لا كتناقض.

---

## المرحلة 4: تنظيف المستخدمين والأدوار والOverrides

**الأهداف:**

1. إزالة الضوضاء من مستخدمي التكامل.
2. تثبيت مصفوفة صلاحيات نظيفة.

**المخرجات:**

1. مستخدم مرجعي واحد لكل دور تشغيلي أساسي.
2. تعطيل/أرشفة الحسابات التكاملية غير اللازمة.
3. تدقيق Overrides غير المبرر.

**اختبارات القبول:**

1. صفحة المستخدمين واضحة وغير مضللة.
2. لا overrides مؤثرة بدون مبرر إداري.

## 9) خطة اختبارات شاملة (Test Strategy)

## 9.1 اختبارات الوحدة (Unit Tests)

### A) اختبارات Policy

1. `ActionabilityPolicyService`:
   1. stage allowed mapping صحيح لكل Permission.
   2. سجل `status!=ready` غير actionable.
   3. سجل `ready + active_action فارغ` لا يصبح actionable للأدوار بعد `data_entry`.
   4. سجل `ready + active_action غير فارغ` يصبح actionable فقط للمرحلة/الدور الصحيح.
   5. سجل `is_locked=true` غير actionable.
   6. رفض السجل يعيد `workflow_step` إلى `draft` ويصفر `active_action`.
   7. رفض السجل بدون ملاحظة => فشل العملية.

### B) اختبارات الرؤية

1. `GuaranteeVisibilityService`:
   1. task-only roles لا ترى إلا actionable.
   2. `ui_full_filters_view` يلغي task-only clamp.
   3. `data_entry/developer` سلوكهم الافتراضي موسع.
   4. صفحات الدفعات لا تظهر إلا لـ `data_entry` (أو استثناء دفعات حوكمي صريح).

### C) اختبارات التنقل

1. `NavigationService`:
   1. أي `filter` لrole تشغيلي يُclamp إلى actionable.
   2. `released` محجوب لغير المخول.
   3. `isIdInFilter` يمنع فتح ID خارج النطاق.

### D) اختبارات إسناد الهوية (Actor Attribution)

1. أي خدمة تسجيل حدث (timeline/notes/attachments):
   1. ترفض إنشاء سجل جديد إذا كانت هوية المنفذ فارغة بالكامل.
   2. تحفظ `actor_display_name` عند توفره.
2. طبقة العرض:
   1. تعرض اسمًا فعليًا للمنفذ حسب ترتيب الأولوية المحدد.
   2. لا تعرض `المستخدم` في الأحداث الجديدة.

## 9.2 اختبارات التكامل (Integration Tests)

### A) مصفوفة أدوار

1. Data Entry:
   1. يمكنه التعامل مع `pending/ready` حسب الصلاحية.
   2. يرى `ready` بدون إجراء ليحدد `extend/reduce/release`.
   3. يستلم `signed + ready` للطباعة.
   4. بعد الطباعة يمكنه بدء إجراء جديد على نفس الضمان دون إنشاء إدخال جديد.
   5. لا يستطيع التعديل أو بدء إجراء جديد إذا كان الضمان `released + signed`.
2. Auditor:
   1. يرى فقط `draft + ready + active_action`.
   2. لا يفتح released أو سجلات خارج مهامه.
   3. عند الرفض تظهر نافذة سبب الرفض، والسبب إلزامي.
3. Analyst/Supervisor/Approver/Signatory:
   1. كل دور يرى مرحلته فقط ضمن actionable.
   2. لا يرى أي سجل `ready` بدون `active_action`.
   3. يملك مسار رفض مع حفظ سبب الرفض في `notes`.
4. Batch Authorization Matrix:
   1. `data_entry` يملك دخول صفحات الدفعات وتنفيذ إجراءاتها.
   2. الأدوار التشغيلية اللاحقة ممنوعة من صفحات الدفعات وAPIs الدفعات.
   3. `developer` يحتاج `batch_full_operations_override` ليستخدم الدفعات تشغيليًا.

### B) فتح مباشر خارج النطاق

1. `GET /index.php?id=<out_of_scope>` => out-of-scope behavior.
2. `GET /api/get-record.php?index=...&filter=all` لrole تشغيلي => يبقى actionable scope.

### C) سلامة الانتقالات

1. `draft -> audited` عند `ready` + `active_action` محدد فقط.
2. منع التقدم عند `pending`.
3. منع التقدم عند `is_locked=true`.
4. منع أي تقدم إذا `active_action` فارغ.
5. أي رفض من أي مرحلة => رجوع إلى `draft + ready + active_action=null`.
6. رفض بدون سبب => مرفوض برمجيًا.
7. بعد `signed` والطباعة: السجل يبقى `ready` ويدعم دورة إجراء جديدة بدون إعادة إدخال.
8. عند `released + signed`: إيقاف أي دورة جديدة ومنع إجراءات `extend/reduce/release`.

### D) lifecycle mutations

1. `extend/reduce/release` لا تعمل إذا `pending` (إلا override policy مقنن).
2. تسجيل timeline صحيح.

### E) تتبع المنفذ End-to-End

1. من `import` حتى `sign/reject/reopen` كل حدث يحمل Actor واضحًا.
2. `notes` و`attachments` تحفظ نفس هوية المنفذ المتوقعة.
3. لا يظهر في السجلات الجديدة أي actor label عام (`المستخدم`).

## 9.3 اختبارات نزاهة البيانات (SQL Integrity Checks)

> تُنفذ يوميًا/قبل الإصدار.

1. خلط اختبار/حقيقي:
```sql
SELECT COUNT(*) AS mixed_batches
FROM (
    SELECT o.batch_identifier
    FROM guarantee_occurrences o
    JOIN guarantees g ON g.id = o.guarantee_id
    GROUP BY o.batch_identifier
    HAVING MIN(COALESCE(g.is_test_data,0)) <> MAX(COALESCE(g.is_test_data,0))
) t;
```

2. ضمانات بدون دفعات:
```sql
SELECT COUNT(*) AS orphan_guarantees
FROM guarantees g
LEFT JOIN guarantee_occurrences o ON o.guarantee_id = g.id
WHERE o.guarantee_id IS NULL;
```

2.1 دفعات metadata بلا occurrence (تنظيف تشغيلي):
```sql
SELECT COUNT(*) AS metadata_without_occurrences
FROM batch_metadata bm
LEFT JOIN guarantee_occurrences o ON o.batch_identifier = bm.import_source
WHERE o.id IS NULL;
```

3. pending خارج draft (للتشخيص):
```sql
SELECT COUNT(*) AS pending_outside_draft
FROM guarantee_decisions d
WHERE d.status = 'pending'
  AND COALESCE(d.workflow_step,'') <> 'draft';
```

4. actionable breaks:
```sql
SELECT COUNT(*) AS invalid_actionable
FROM guarantee_decisions d
WHERE d.status = 'ready'
  AND COALESCE(d.active_action,'') = ''
  AND COALESCE(d.is_locked,false) = false
  AND COALESCE(d.workflow_step,'') NOT IN ('draft','audited','analyzed','supervised','approved');
```

## 17) ملحق A - تعريفات تشغيلية مختصرة

1. `draft`: مرحلة إدخال/تدقيق أولي.
2. `audited`: تم التدقيق.
3. `analyzed`: تم التحليل.
4. `supervised`: تم إشراف التحليل.
5. `approved`: تم الاعتماد.
6. `signed`: تم التوقيع.
7. `pending`: بيانات ناقصة.
8. `ready`: بيانات مكتملة وجاهزة تشغيليًا.
9. `locked`: سجل مقفل تشغيليًا.
10. `actionable`: سجل قابل لتنفيذ الإجراء الآن.
11. `action-gated workflow`: لا تبدأ دورة الاعتماد إلا بعد اختيار `extend/reduce/release` من `data_entry`.
12. `reject`: رفض من أحد أدوار الاعتماد يعيد السجل إلى `data_entry` مع سبب إلزامي.
13. `rework`: إعادة عمل لدى `data_entry` بعد رفض موثق.
14. `finalized (released+signed)`: حالة نهائية مغلقة، بلا تعديل أو إجراء جديد.

3. **Analyst**: مسؤول تحليل.

## 20) ملحق D - تعريف "منتهي"

لا تعتبر خطة التصحيح منتهية حتى تحقق:

1. Zero critical drift.
2. Zero mixed test/real data.
3. Zero unauthorized out-of-scope access.
4. توافق كامل بين العدادات الأساسية.
5. استقرار اختبارات الوحدة والتكامل.
