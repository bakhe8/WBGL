# سياسة التنفيذ المعتمد على Docs — WBGL

تاريخ الإصدار: 2026-02-28  
الحالة: **Active / Mandatory**  
الغرض: اعتماد `Docs` كمحرك القرار والتنفيذ الإلزامي لكل العمل المستقبلي في المشروع.

---

## 1) القرار الرسمي

- `Docs` هو المصدر الأساسي للقرار قبل أي تغيير تقني أو تشغيلي.
- لا يبدأ أي تنفيذ جديد قبل الرجوع إلى الوثائق المرجعية المعتمدة في `Docs/INDEX-DOCS-AR.md`.
- عند تعارض أي ملف خارج `Docs` مع وثائق `Docs`، تكون الأولوية دائمًا لـ `Docs`.

---

## 2) الطبقة المرجعية الإلزامية

المراجع الحاكمة قبل التنفيذ:
1. `Docs/WBGL-UNIFIED-RESOLVED-REPORT-AR-2026-02-28.md`
2. `Docs/WBGL-FINAL-CLAIMS-REGISTER-AR-2026-02-28.md`
3. `Docs/WBGL-UNIFIED-OWNER-VISION-AR-2026-02-28.md`
4. `Docs/WBGL-FULL-TRACEABILITY-MATRIX-AR.md`
5. `Docs/INDEX-DOCS-AR.md`

---

## 3) بروتوكول التنفيذ (Mandatory Workflow)

قبل أي تنفيذ:
1. تحديد البنود المتأثرة (`FR/R/G/C`) من المصفوفة.
2. توثيق قرار التنفيذ وسببه والـ scope المتوقع.
3. التأكد أن التغيير لا يكسر Guardrails الحوكمة/الأمن.

أثناء التنفيذ:
1. ربط كل PR/Commit ببنود `FR/R/G/C` واضحة.
2. منع التعديلات غير الموثقة في النطاق.
3. أي تغيير حساس يمر عبر `change record` قابل للتدقيق.

بعد التنفيذ:
1. تحديث الوثيقة/الوثائق المتأثرة في `Docs`.
2. تحديث حالة البنود في `WBGL-FULL-TRACEABILITY-MATRIX-AR.md`.
3. منع إغلاق أي مرحلة بدون دليل تغطية محدث.

التنفيذ المتتابع الإجباري:
1. تشغيل `php app/Scripts/sequential-execution.php status` قبل بدء أي عمل.
2. تنفيذ الخطوة الوحيدة ذات الحالة `in_progress` فقط.
3. إغلاق الخطوة عبر `complete` مع `evidence` و`refs` قبل الانتقال لما بعدها.
4. فشل `php app/Scripts/sequential-execution.php guard` يعني منع الإطلاق/الدمج.

تجميد الميزات أثناء الاستقرار (Phase-1 Freeze):
1. عند وجود خطوة نشطة من نوع `P1-*` يتم تفعيل Gate تجميد الميزات تلقائيًا.
2. أي PR بمسارات حساسة يجب أن يحتوي `CHANGE-TYPE` و`STABILITY-REFS` مرتبطًا بخطوات وتغطيات `P1`.
3. يمنع إضافة أسطح ميزات جديدة أثناء `P1-*` (`api/`, `views/`, `templates/`, `partials/`, `public/js/`, `public/css/`).
4. مرجع القرار والتنفيذ: `Docs/WBGL-STABILITY-FREEZE-GATE-AR.md`.

---

## 4) قواعد الإلزام

1. لا ميزات جديدة قبل إغلاق بنود الاستقرار الحرجة المعتمدة.
2. أي بند يحقق Target ويكسر Guardrail يعود إلى `Open`.
3. لا انتقال بين المراحل بدون مرور `Traceability Gate`.
4. لا اعتماد نهائي لأي إنجاز إذا لم ينعكس في `Docs`.
5. لا دمج لتغييرات وظيفية خارج Scope الاستقرار أثناء `P1-*`.

---

## 5) مسؤوليات الفرق

1. مالك النظام: اعتماد الأولويات والحسم النهائي عند التعارض.
2. القائد التقني: ضمان التزام التنفيذ بالمراجع الحاكمة.
3. QA/Compliance: التحقق من التغطية والدليل قبل الإغلاق.
4. كل مساهم: تحديث `Docs` كجزء إلزامي من أي تغيير.

---

## 6) معيار النجاح

- نجاح السياسة يعني أن `Docs` يعكس دائمًا الواقع التنفيذي الحالي.
- أي فرق بين التنفيذ والوثائق يعتبر Incident حوكمي يجب إغلاقه.
